# 高并发库存扣减设计：从防超卖到最终一致性

库存链路的第一优先级是稳定和正确，性能优化必须服务于这个目标。对电商秒杀场景来说，核心问题不是“如何更快扣库存”，而是“如何在峰值流量下不超卖、不拖垮主链路，并能把异常闭环补偿回来”。

## 场景分析

典型请求路径可以拆成四段：

1. 网关和活动层做准入校验，拦截无效用户、无效活动和明显超额流量。
2. Redis 承接热点库存预扣减，避免所有流量直接打到数据库。
3. 下单请求写入 MQ，由订单应用异步消费创建订单。
4. 数据库使用乐观锁做最终兜底，失败后触发库存回补和订单关闭。

这个设计不追求强同步体验，而是用最终一致性换取主链路抗压能力。

## 流量模型假设

| 指标 | 假设值 | 设计含义 |
| --- | --- | --- |
| 峰值 QPS | 50,000 | 入口必须限流，不能让数据库承接峰值 |
| 单 SKU 库存 | 10,000 | Redis 需要承担热点 Key 压力 |
| 下单成功率 | 20% 以下 | 大部分请求应该在前置阶段被过滤 |
| 可接受一致性窗口 | 秒级 | 适合 MQ 异步削峰和补偿对账 |

## 架构设计

核心原则是把库存链路拆成“预扣减”和“确认扣减”两阶段：

- Redis 负责高并发预扣减，使用 Lua 保证检查和扣减原子性。
- MQ 负责削峰，订单创建和数据库扣减异步执行。
- MySQL 负责库存事实表，使用 `where stock >= amount and version = ?` 或 `stock >= amount` 的条件更新兜底。
- 定时任务负责对账，扫描 Redis、订单、库存流水之间的不一致数据。

## 并发控制方案

Redis 预扣减脚本需要做到原子校验：

```lua
local stock = tonumber(redis.call('get', KEYS[1]))
local amount = tonumber(ARGV[1])
if stock == nil then
  return -1
end
if stock < amount then
  return 0
end
redis.call('decrby', KEYS[1], amount)
return 1
```

数据库层使用乐观更新：

```sql
update sku_stock
set available_stock = available_stock - #{amount},
    version = version + 1,
    update_time = now()
where sku_id = #{skuId}
  and available_stock >= #{amount}
  and version = #{version}
```

线程安全边界需要明确：应用内不要用本地锁保护库存正确性，本地锁只能用于单机热点保护；跨实例正确性必须依赖 Redis 原子操作、数据库条件更新和幂等流水。

## 风险与兜底

- Redis 扣减成功但 MQ 发送失败：本地事务表或可靠消息表记录待发送事件，失败后补偿回补库存。
- MQ 重复消费：订单号、用户活动维度、库存流水号必须做幂等唯一约束。
- 数据库扣减失败：关闭订单并回补 Redis，记录失败原因用于对账。
- Redis 和数据库库存不一致：以数据库事实表和库存流水为准，定时修正 Redis 缓存。

## 监控与报警

必须覆盖这些指标：

- 活动入口 QPS、限流量、拒绝率。
- Redis 预扣减成功率、失败率、Lua 执行耗时。
- MQ 积压量、消费 RT、重试次数、死信数量。
- 数据库库存扣减成功率、乐观锁失败率、慢 SQL。
- 库存对账差异量、补偿成功率、补偿失败明细。

## 演进方向

前期可以用单 Redis Key 承接库存；当热点 SKU 压力过高时，再演进为库存分段。分段后要注意两个问题：一是分段分配不能造成局部库存提前耗尽，二是对账任务需要聚合所有分段库存，避免排查成本失控。

整体演进顺序建议是：单 Key 预扣减、MQ 异步落库、库存流水、对账补偿、热点 SKU 分段、活动级隔离和多机房容灾。
